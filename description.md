# Описание курсовой работы — Онлайн-магазин книг

## 1. ER-модель

ER-модель описывает предметную область на концептуальном уровне.

---

### USER — Пользователь

Центральная сущность: всё в магазине делается для пользователей и от их имени.

| Атрибут | Описание |
|---|---|
| **UserId** | Суррогатный идентификатор — просто порядковый номер. Сам по себе не несёт смысла, но нужен как стабильная ссылка на пользователя из других мест (заказы, отзывы, сессии). |
| **Email** | Адрес электронной почты — уникален в системе, через него пользователь входит в аккаунт. Служит альтернативным идентификатором: если нужно найти пользователя по email, это работает так же надёжно, как и по UserId. |
| **FullName** | Имя и фамилия. Отображается в интерфейсе, на страницах заказов, в квитанциях. |
| **RegisteredAt** | Дата и время регистрации. Используется для сортировки новых пользователей, в маркетинговой аналитике («сколько пользователей зарегистрировалось за этот месяц»). |
| **IsActive** | Флаг активности аккаунта (true/false). Если пользователь нарушил правила, его можно заблокировать, не удаляя — история заказов и отзывов при этом сохраняется. |

---

### AUTHOR — Автор книги

Авторы книг — самостоятельная сущность. Автор существует в системе независимо от того, есть ли его книги в каталоге прямо сейчас.

| Атрибут | Описание |
|---|---|
| **AuthorId** | Суррогатный идентификатор автора. |
| **FullName** | Полное имя автора — обязательно, это основной способ найти и отобразить автора. |
| **Biography** | Краткая биография для карточки автора. Необязательна — не у всех авторов есть подробная информация, и магазин работает без неё. |
| **BirthDate** | Дата рождения. Необязательна — полезна для исторических и классических авторов, но не всегда известна. |
| **Country** | Страна происхождения. Помогает фильтровать книги по стране автора. Необязательна. |

---

### CATEGORY — Категория

Категории организуют книжный каталог. Ключевая особенность — иерархическая структура: категория может входить в другую категорию.

| Атрибут | Описание |
|---|---|
| **CategoryId** | Суррогатный идентификатор категории. |
| **ParentCategoryId** | Ссылка на родительскую категорию — именно через этот атрибут строится иерархия («Фантастика» входит в «Художественную литературу»). У корневых категорий это поле пустое (NULL). |
| **Name** | Название категории, которое видит покупатель: «Детективы», «Учебники», «Биографии». |

Рекурсивная связь CATEGORY → CATEGORY позволяет строить дерево любой глубины без изменений структуры базы данных.

---

### BOOK — Книга

Главная сущность каталога — именно ради книг существует весь магазин.

| Атрибут | Описание |
|---|---|
| **BookId** | Суррогатный идентификатор книги. |
| **ISBN** | Международный стандартный номер книги — уникален для каждого издания. Это естественный идентификатор книги в реальном мире, поэтому он тоже может однозначно идентифицировать запись (альтернативный ключ). |
| **Title** | Название книги. Обязательно и используется везде: в поиске, каталоге, заказах. |
| **Description** | Аннотация — краткое описание содержания. Необязательна: некоторые книги добавляются в каталог без описания. По этому полю работает полнотекстовый поиск. |
| **PublicationYear** | Год публикации. Необязателен: для старых изданий он может быть неизвестен. Помогает фильтровать книги по периоду. |
| **Pages** | Количество страниц. Необязательно: используется для фильтрации («только книги до 200 страниц»). |
| **Price** | Текущая цена книги. Обязательна — без неё нельзя продать товар. Важно: это *текущая* цена, а не историческая — цена в конкретном заказе хранится отдельно в ORDER_ITEM. |
| **Tags** | Список меток: «классика», «бестселлер», «школьная программа», «экранизирован». Один из инструментов фильтрации — покупатель может выбрать сразу несколько тегов. Хранится как массив строк. |
| **Metadata** | Дополнительные сведения в свободном формате: издательство, язык, номер издания, литературные награды. Разные книги могут иметь разный набор этих полей, поэтому хранится как JSONB. |

---

### WAREHOUSE — Склад

Магазин может работать с несколькими складами в разных городах или регионах.

| Атрибут | Описание |
|---|---|
| **WarehouseId** | Суррогатный идентификатор склада. |
| **Name** | Название склада: «Москва-Север», «Санкт-Петербург», «Новосибирск». |
| **Location** | Географические координаты (широта и долгота). Нужны, чтобы найти ближайший к покупателю склад и рассчитать стоимость доставки. |
| **Capacity** | Вместимость склада в единицах товара. Используется для планирования логистики. |

---

### ORDER — Заказ

Создаётся в момент, когда покупатель оформляет корзину.

| Атрибут | Описание |
|---|---|
| **OrderId** | Суррогатный идентификатор заказа. |
| **UserId** | Ссылка на покупателя. Обязательна — анонимных заказов нет. |
| **Status** | Текущее состояние заказа в жизненном цикле: `new` → `paid` → `shipped` → `completed` или `cancelled`. |
| **TotalAmount** | Итоговая сумма заказа. Хранится прямо здесь, а не вычисляется из позиций — во-первых, это быстрее при отображении истории заказов; во-вторых, цены книг со временем меняются, а сумма заказа должна оставаться неизменной. |
| **CreatedAt** | Дата и время оформления заказа. Заказы вставляются строго в хронологическом порядке — это критично для эффективности BRIN-индекса. |
| **UpdatedAt** | Дата и время последнего изменения статуса заказа. |

---

### ORDER_ITEM — Позиция заказа

Каждая строка в этой сущности — одна книга в одном заказе. Позиция заказа не имеет смысла без самого заказа, поэтому это зависимая сущность.

| Атрибут | Описание |
|---|---|
| **OrderItemId** | Суррогатный идентификатор позиции. Упрощает ссылки на конкретную строку заказа (например, для возвратов). |
| **OrderId** | Ссылка на заказ, к которому относится позиция. |
| **BookId** | Ссылка на книгу. |
| **Quantity** | Количество экземпляров этой книги в заказе. Минимум 1. |
| **UnitPrice** | Цена книги *на момент покупки*. Это историческая цена — она фиксируется при оформлении заказа и больше не меняется, даже если в каталоге цена на книгу впоследствии изменится. |

---

### REVIEW — Отзыв

Покупатель может оставить отзыв о прочитанной книге. Отзыв принадлежит и конкретному пользователю, и конкретной книге одновременно.

| Атрибут | Описание |
|---|---|
| **ReviewId** | Суррогатный идентификатор отзыва. |
| **UserId** | Кто написал отзыв. |
| **BookId** | О какой книге отзыв. |
| **Rating** | Оценка от 1 до 5 звёзд. Обязательна — без оценки отзыв бессмысленен. |
| **ReviewText** | Текст отзыва. Необязателен — пользователь может поставить только оценку без комментария. По этому полю работает полнотекстовый поиск. |
| **IsPublished** | Флаг публикации. По умолчанию `true`, но можно выставить `false` для модерации — отзыв скрывается из каталога, но не удаляется. |
| **CreatedAt** | Дата и время написания отзыва. |

---

### USER_SESSION — Сессия пользователя

Хранит информацию о каждом входе в систему. Используется для безопасности и аналитики.

| Атрибут | Описание |
|---|---|
| **SessionId** | Суррогатный идентификатор сессии. |
| **UserId** | Какой пользователь вошёл. |
| **IP** | IP-адрес устройства. Хранится в специальном типе `inet`, который понимает как IPv4, так и IPv6, и умеет работать с подсетями (например, «все сессии из подсети 10.0.0.0/8»). |
| **StartedAt** | Дата и время начала сессии. |
| **UserAgent** | Браузер и операционная система (строка вида «Mozilla/5.0 ... Chrome/120»). Необязателен — полезен для аналитики устройств. |

---

### Ассоциативные сущности

**BOOK_AUTHOR** — разрешает связь «многие ко многим» между книгами и авторами. У одной книги может быть несколько авторов (соавторы, переводчики), и один автор может написать несколько книг. Хранит только пару (BookId, AuthorId).

**BOOK_CATEGORY** — то же самое для книг и категорий. Одна книга может относиться сразу к нескольким категориям (например, «биография» и «история»), и в одной категории может быть множество книг.

**STOCK** — связь книги со складом с дополнительным атрибутом: помимо самого факта связи хранит **Quantity** — количество экземпляров конкретной книги на конкретном складе. Без этого атрибута сущность была бы просто «книга хранится на складе», а нам нужно знать «сколько».

---

### Связи и их обоснование

| Связь | Кардинальность | Почему именно так |
|---|---|---|
| USER → ORDER | 1 : 0..N | Пользователь может только зарегистрироваться и не сделать ни одного заказа. Один заказ принадлежит ровно одному пользователю. |
| ORDER → ORDER_ITEM | 1 : 1..N | Пустого заказа не бывает — минимум одна позиция обязана быть. |
| BOOK → ORDER_ITEM | 1 : 0..N | Новинка может быть в каталоге, но ещё не куплена никем. |
| USER → REVIEW | 1 : 0..N | Пользователь может не писать отзывов вообще. |
| BOOK → REVIEW | 1 : 0..N | Книга может не иметь ни одного отзыва — например, только что добавлена. |
| USER → USER_SESSION | 1 : 0..N | Один пользователь входит с разных устройств и браузеров. |
| BOOK ↔ AUTHOR | M : N | Один автор — много книг; одна книга — несколько авторов. Разрешается через BOOK_AUTHOR. |
| BOOK ↔ CATEGORY | M : N | Книга может попадать сразу в несколько категорий. Разрешается через BOOK_CATEGORY. |
| BOOK ↔ WAREHOUSE | M : N | Книга хранится на нескольких складах; на каждом складе — много книг. Разрешается через STOCK. |
| CATEGORY → CATEGORY | 0..1 : 0..N | Рекурсивная иерархия. Корневые категории не имеют родителя (NULL). |
| BOOK → BOOK_AUTHOR | 1 : 1..N | Книга обязана иметь хотя бы одного автора. |

## 2. Модель семантических объектов (SOM)

SOM — альтернативный способ описать предметную область. Вместо сущностей и связей здесь «объекты» с атрибутами, у каждого из которых задана кардинальность **min.max**.

**Что означает кардинальность:**
- `1.1` — атрибут обязателен и всегда один (имя пользователя, цена книги)
- `0.1` — атрибут может отсутствовать, но если есть — то только одно значение (биография автора)
- `0.N` — атрибут может отсутствовать и может быть множественным (теги книги)
- `1.N` — атрибут обязателен и может быть множественным (авторы книги — хотя бы один)

**Типы объектов по классификации Кренке–Амос:**

| Тип | Объекты | Характеристика |
|---|---|---|
| Простой | WAREHOUSE | Все атрибуты — простые скалярные значения, нет ссылок на другие объекты |
| Составной | USER, AUTHOR, CATEGORY, REVIEW, USER_SESSION | Содержит ссылки на другие объекты (например, REVIEW ссылается на USER и BOOK) |
| Гибридный | BOOK, ORDER | Содержит и многозначные атрибуты (теги книги, позиции заказа), и ссылки |
| Ассоциативный | STOCK | Сам по себе идентифицируется только парой двух объектов (BOOK + WAREHOUSE) |

**Особые атрибуты:**

**BOOK.Tags (`0.N`)** — у книги может не быть тегов, а может быть сразу пять. Это многозначный скалярный атрибут. В реляционной модели реализован как массив `text[]` — специально ради GIN-индекса, который позволяет мгновенно находить все книги с нужным тегом.

**BOOK.Metadata (`0.1`)** — групповой атрибут: внутри него несколько полей (издатель, язык, номер издания, награды). Разные книги могут иметь разный набор этих полей, поэтому хранится как JSONB.

**ORDER.LineItem (`1.N`)** — каждая позиция заказа — это тройка (книга + количество + цена). Это гибридный атрибут: в реляционной модели он выносится в отдельную таблицу ORDER_ITEMS.

## 3. Реляционная модель

### Первичные ключи

Почти везде используется **суррогатный PK** — автоинкрементный `BIGSERIAL`. Причины:

- Email пользователя может измениться — тогда все ссылки из других таблиц пришлось бы обновлять. С числовым id этой проблемы нет.
- Числовой ключ компактнее строки в памяти и быстрее при JOIN.
- Монотонно возрастающие значения — оптимальны для B-tree-индекса.

Исключение — **BOOK_AUTHORS, BOOK_CATEGORIES, STOCK**: здесь смысл записи — это и есть пара (книга + автор). Суррогатный ключ был бы искусственным и избыточным, поэтому используется составной PK из двух FK.

### Нормальные формы

Все таблицы приведены к **третьей нормальной форме (3НФ)**: каждый атрибут зависит только от первичного ключа, а не от других атрибутов. Ассоциативные таблицы без собственных атрибутов находятся в **BCNF**.

Два осознанных отступления от строгой нормализации:
- **BOOKS.Tags** хранится как массив `text[]` вместо отдельной таблицы — ради GIN-индекса для быстрого поиска по тегам.
- **ORDERS.TotalAmount** хранится в готовом виде вместо вычисления через SUM(ORDER_ITEMS) — ради скорости отображения истории заказов и сохранения исторической суммы.

### Каскадные правила

**ON DELETE CASCADE** применяется там, где дочерние записи без родителя теряют смысл:
- Удалили пользователя → его сессии и отзывы тоже удаляются.
- Удалили заказ → его позиции тоже удаляются.
- Удалили книгу → её записи в BOOK_AUTHORS, BOOK_CATEGORIES, STOCK, REVIEWS тоже удаляются.

**ON DELETE RESTRICT** — там, где удаление родителя опасно:
- Нельзя удалить пользователя, если у него есть заказы (это потеря финансовой истории).
- Нельзя удалить книгу, если она фигурирует в позициях заказов.

**ON DELETE SET NULL** — для иерархии категорий:
- Если удалить категорию «Фантастика», её подкатегории («Космическая опера», «Киберпанк») не удаляются — у них просто обнуляется ссылка на родителя, и они становятся корневыми. Лучше потерять иерархию, чем весь контент.

### Генерируемый столбец fts

Столбец `fts` в таблице BOOKS — это вектор полнотекстового поиска, автоматически вычисляемый из названия и описания книги. Заголовок получает больший вес (A), описание — меньший (B). PostgreSQL пересчитывает столбец сам при каждом изменении `title` или `description` — приложению не нужно об этом думать. По `fts` построен GIN-индекс, который делает поиск по всему каталогу мгновенным.

## 4. Программная часть

### sql/schema.sql — Схема базы данных

Создаёт 12 таблиц в правильном порядке (сначала независимые сущности, потом зависимые). Перед созданием таблиц включаются три расширения:

- `pg_trgm` — поддержка триграммного поиска (нечёткий поиск по подстроке, опечатки)
- `btree_gin` — позволяет строить GIN-индексы на типах, которые обычно индексируются только B-tree
- `btree_gist` — аналогично для GiST

### sql/seed.sql — Тестовые данные

Заполняет базу синтетическими данными в объёмах, при которых разница между типами индексов хорошо видна на замерах:

| Таблица | Строк | Зачем столько |
|---|---|---|
| users | 1 000 000 | Высокая кардинальность email для Hash/B-tree |
| books | 500 000 | Достаточно для FTS и GIN по тегам |
| orders | 2 000 000 | Большой временной ряд для BRIN vs B-tree |
| order_items | ~3 000 000 | Зависят от orders (1–2 позиции на заказ) |
| user_sessions | 2 000 000 | Для SP-GiST по IP |
| reviews | 500 000 | Для GIN полнотекстового поиска |
| warehouses | 20 | Небольшой справочник |

Важная деталь: заказы вставляются строго в хронологическом порядке. BRIN-индекс работает эффективно только тогда, когда физический порядок строк на диске совпадает с порядком значений в индексируемом столбце. Если вставлять заказы вразброс по датам, BRIN не даст выигрыша.

### sql/indexes.sql — Индексы

Создаёт 18 индексов шести типов. Каждый тип применяется там, где он действительно полезен:

| Тип | Где применяется | Почему |
|---|---|---|
| B-tree | orders.created_at, orders.status, books.price, reviews.rating, users.registered_at | Универсальный индекс: равенство, диапазоны, сортировка |
| Hash | orders.user_id, order_items.book_id | Только точечный поиск по равенству, но компактнее B-tree |
| GiST | warehouses.location, books.title (триграммы), reviews.text (триграммы) | Геопоиск и нечёткий поиск по подстроке |
| SP-GiST | user_sessions.ip, warehouses.location | IP-подсети (оператор `<<`), альтернатива GiST на несбалансированных пространствах |
| GIN | books.fts, books.tags, books.metadata, reviews.fts | Полнотекстовый поиск, массивы (`@>`), JSONB (`@>`, `?`) |
| BRIN | orders.created_at, user_sessions.started_at | Монотонные временны́е ряды: почти нет накладных расходов, крошечный размер |

### collect_bench/main.go — Сборщик бенчмарков

Go-программа автоматизирует сравнение производительности индексов.

**Проблема с ручными замерами:** если просто запустить запрос и посмотреть время, PostgreSQL сам выберет наилучший план — и сравнение «B-tree vs Hash» не получится, потому что оптимизатор выберет один из них. Программа обходит это через `SET enable_seqscan=off`, `SET enable_indexscan=off` и другие настройки — принудительно включая нужный план для каждого варианта.

**Почему медиана, а не среднее:** первый запрос часто медленнее из-за холодного кэша, а случайный фоновый процесс может дать выброс. Медиана из 3–5 прогонов устойчива к таким выбросам.

**Сценарии:**
1. B-tree vs BRIN на `orders.created_at` — диапазонный запрос по датам
2. Hash vs B-tree на `orders.user_id` — точечный поиск заказов пользователя
3. GIN vs Seq Scan на `books.fts` — полнотекстовый поиск по каталогу
4. GIN vs Seq Scan на `books.tags` — поиск книг с нужным тегом (`@>`)
5. SP-GiST vs Seq Scan на `user_sessions.ip` — поиск сессий из IP-подсети

Результаты записываются в `frontend/data/results.json`, который подхватывает веб-интерфейс.

Используется `pgx/v5` — нативный драйвер PostgreSQL для Go. В отличие от стандартного `database/sql`, он работает напрямую по протоколу PostgreSQL без промежуточного слоя абстракции, что чуть быстрее и даёт доступ к специфичным для PostgreSQL возможностям.

### frontend/ — Веб-интерфейс

Написан на чистом HTML + CSS + JavaScript, без фреймворков. Единственная внешняя зависимость — **Mermaid.js** для рендеринга диаграмм, загружается с CDN.

Пять вкладок:

**ER-модель** — интерактивная диаграмма сущностей и связей. Кнопки +/−/⟳ для масштабирования, перетаскивание мышью для навигации. Диаграмма генерируется динамически из JS-массива ENTITIES и строится через `mermaid.render()`.

**Семантическая модель** — карточки для каждого объекта: тип объекта по SOM, список атрибутов с кардинальностями (min.max) и пояснениями.

**Реляционная модель** — Mermaid-диаграмма всех 12 таблиц с атрибутами (4 колонки: атрибут / тип / ключ / null) + детальные карточки каждой таблицы (5 колонок: Название / Тип / Ключ / NULL / Примечание) + таблица типов связей + таблицы правил минимальной кардинальности.

**Типы индексов** — карточки для шести типов (B-tree, Hash, GiST, SP-GiST, GIN, BRIN) с описанием внутренней структуры и типичных сценариев, плюс сравнительная матрица по скорости равенства, диапазона, сортировки и размеру индекса.

**Бенчмарки** — сравнительная матрица производительности и реальные результаты из `results.json`. Вкладка делает `fetch('data/results.json')` — именно поэтому нужен HTTP-сервер, а не открытие `file://` напрямую.
